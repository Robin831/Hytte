# Stride — AI Running Coach

**Status**: Planning  
**Created**: 2026-04-05  

## Overview

Stride is an AI running coach that lives inside the Training section of Hytte. It generates weekly training plans based on the user's goals, fitness data, workout history, and personal notes. It continuously monitors progress and adjusts guidance as new workouts come in.

The name **Stride** evokes forward momentum, consistency, and the fundamental unit of running — each stride building toward the goal.

## Training Philosophy — Marius Bakken Model

Stride follows the Norwegian Marius Bakken training model. Key principles:

- **Threshold is king**: For recreational runners doing 3-5 sessions/week, threshold work dominates. This is NOT 80/20 polarized (that's for elite athletes doing 10+ sessions/week).
- **Threshold means BELOW threshold**: All threshold sessions target just below the lactate threshold — never on or above it. Controlled, sustainable effort.
- **One hard session per week (or every other week)**: The only time intensity goes above threshold is one specific session — e.g., 45s hard + 15s rest × 20-40 reps in various forms, or hill intervals. This is the single "spice" session.
- **Typical week structure (3-4 sessions)**: 1 threshold, 1 long run with threshold elements, 1-2 easy/recovery, and optionally the hard session every 1-2 weeks.
- **Typical week structure (5 sessions)**: 2 threshold, 1 long run, 1 easy, 1 hard (or easy if not a hard week).
- **Long run day**: Sunday (default).

## Core Concepts

### Weekly Plan Cycle

```
Sunday 18:00 → Stride generates next week's plan
  ↓
Mon-Sun → User trains, uploads workouts
  ↓
Nightly 03:00 → Stride evaluates new workouts against plan
  ↓
Sunday 18:00 → Next plan incorporates feedback + adjustments
```

### Goal Hierarchy

1. **Primary Goal** — one big target race/event (e.g., Bergen City Marathon, Oct 2026, sub-3:30)
2. **Secondary Races** — smaller races along the way that serve as fitness checks or tune-ups
3. **Training Blocks** — mesocycles that Stride structures around goal dates (base, build, peak, taper)

### Data Available to Stride

Everything the existing AI analysis already has access to, plus:
- All past weekly plans and their completion rates
- User notes/comments per plan (free text input)
- Race calendar with A/B/C priority races
- Training block context (what phase we're in)
- Lactate test data and zone boundaries
- VO2max estimates and trend
- ACR (acute:chronic ratio) for injury risk
- Custom user prompt (from settings, like the AI analysis prompt)

## Data Model

### New Tables

```sql
-- The race calendar: upcoming events with priority
CREATE TABLE stride_races (
    id          INTEGER PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name        TEXT NOT NULL,       -- encrypted
    date        TEXT NOT NULL,       -- YYYY-MM-DD
    distance_m  REAL NOT NULL,       -- meters
    target_time INTEGER,             -- seconds, nullable
    priority    TEXT NOT NULL DEFAULT 'B',  -- 'A' (primary goal), 'B' (important), 'C' (tune-up)
    notes       TEXT NOT NULL DEFAULT '',   -- encrypted
    result_time INTEGER,             -- actual result, nullable
    created_at  TEXT NOT NULL DEFAULT ''
);

-- Weekly training plans generated by Stride
CREATE TABLE stride_plans (
    id          INTEGER PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    week_start  TEXT NOT NULL,       -- YYYY-MM-DD (Monday)
    week_end    TEXT NOT NULL,       -- YYYY-MM-DD (Sunday)
    phase       TEXT NOT NULL DEFAULT '',    -- base, build, peak, taper, recovery
    plan_json   TEXT NOT NULL,       -- encrypted JSON: structured plan with daily sessions
    prompt      TEXT NOT NULL DEFAULT '',    -- encrypted: the prompt sent to Claude
    response    TEXT NOT NULL DEFAULT '',    -- encrypted: raw Claude response
    model       TEXT NOT NULL DEFAULT '',
    created_at  TEXT NOT NULL DEFAULT '',
    UNIQUE(user_id, week_start)
);

-- User notes that feed into the next plan generation
CREATE TABLE stride_notes (
    id          INTEGER PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    plan_id     INTEGER REFERENCES stride_plans(id) ON DELETE SET NULL,
    content     TEXT NOT NULL,       -- encrypted
    created_at  TEXT NOT NULL DEFAULT ''
);

-- Nightly evaluations: Stride's commentary on completed workouts vs plan
CREATE TABLE stride_evaluations (
    id          INTEGER PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    plan_id     INTEGER NOT NULL REFERENCES stride_plans(id) ON DELETE CASCADE,
    workout_id  INTEGER REFERENCES workouts(id) ON DELETE SET NULL,
    eval_json   TEXT NOT NULL,       -- encrypted JSON: workout-vs-plan comparison
    created_at  TEXT NOT NULL DEFAULT ''
);
```

## Scheduling

### Weekly Plan Generation — Sundays 18:00 (Europe/Oslo)

1. Gather all context:
   - User profile (zones, max HR, threshold, VO2max)
   - Race calendar (upcoming races, weeks until primary goal)
   - Last 8 weeks of training history (volume, intensity, types)
   - Current week's plan + completion status
   - Current week's evaluations
   - User notes from the current week
   - ACR and training status
   - Custom user prompt from settings
2. Build prompt with plan structure schema
3. Call Claude (uses model from user's claude_model setting, default claude-opus-4-6)
4. Parse response into plan_json
5. Store in stride_plans
6. Push notification: "Stride has your plan for next week"

### Nightly Evaluation — Daily 03:00 (Europe/Oslo)

1. Find today's completed workouts that haven't been evaluated
2. Match each workout to the planned session (by date + type)
3. Build evaluation prompt with:
   - The planned session details
   - Actual workout data (laps, HR, pace, distance)
   - Weekly context (what's done so far, what's coming)
4. Call Claude for each new workout
5. Store evaluation
6. If critical flag (injury risk, significant deviation): push notification

### Workout Matching

When a workout is uploaded, match it to the plan:
- Same date → direct match
- ±1 day and similar type → fuzzy match (show "Stride expected this on Thursday")
- Unmatched → bonus workout, Stride comments on it in evaluation

### Notes Field

Below the weekly plan, a simple text input:
- "Feeling tired in the legs this week"
- "Want to do trail instead of road on Saturday"
- "Left knee slightly sore after Thursday"
- Notes are included in the next plan generation prompt

## Prompt Design

### Plan Generation Prompt (Sunday)

```
You are Stride, an experienced running coach following the Marius Bakken training
model. You create highly specific weekly training plans tailored to the athlete's
current fitness, goals, fatigue level, and feedback.

## Training Philosophy (Marius Bakken Model)
- For recreational runners (3-5 sessions/week), threshold work is the dominant
  training stimulus — NOT 80/20 polarized (that's for elite 10+ sessions/week).
- "Threshold" means BELOW lactate threshold — controlled, sustainable, never on
  or above it. If the athlete's threshold HR is 172, target 165-170.
- Only ONE session per week (or every other week) goes above threshold: specific
  hard sessions like 30-45s hard + 15s rest × 20-40 reps, or hill intervals.
- Long runs often include threshold elements in the second half (progressive finish).

## Session Specificity Requirements
Every session must include:
- Exact pace ranges derived from threshold pace, adjusted for current fatigue/load
- HR caps (never just "easy" — specify "HR below 150" or "Z1-Z2, 130-150 bpm")
- For intervals: exact rep count, work duration, target pace AND HR, rest duration and effort
- Warmup and cooldown with pace and HR targets
- Strides: count, duration, pace, recovery format
- Contextual notes: when to back off, what to prioritize, form cues

## Athlete Profile
{user_profile_block}   ← existing BuildUserProfileBlock()

## Primary Goal
{goal_race details + weeks until}

## Race Calendar
{upcoming races with priorities}

## Current Training Status
{ACR, training status, VO2max trend, weekly_load trend}

## Last 8 Weeks
{weekly_summaries table}

## Last Week's Plan & Completion
{plan_json from last week + which sessions were completed + evaluations}

## Athlete Notes
{stride_notes from current week}

## Custom Coach Instructions
{stride_custom_prompt from settings}

Generate next week's training plan as JSON following this schema:
{plan_json_schema}

Important:
- Respect the athlete's available training days
- Never exceed the weekly distance cap
- Follow the Marius Bakken model: threshold-dominant for recreational runners
- Threshold sessions MUST target below threshold HR — NEVER on or above
- Only ONE session per week (or every 2 weeks) is allowed above threshold
- Build volume gradually (max 10% increase per week)
- Include one rest day minimum
- Every session must have specific targets: pace ranges (based on threshold), HR caps, interval structure
- Warmup/cooldown paces and HR zones must be explicit
- Strides should specify count, duration, pace, and recovery
- Coach notes should reference last week's performance metrics and this week's focus
- Account for current fatigue (ACR, recent training load) when setting pace targets
```
